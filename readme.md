# 主数据平台

集团统一管主数据的地方。机构、科室、员工、药品、耗材、诊疗项目、设备、医嘱、物资、规范这些数据，都从这里进、从这里出。

代码在 `mdm-platform/`，技术细节看 [tech-doc.md](./tech-doc.md)。

---

## 为啥要做

集团现在 30 家机构，HIS、EMR、ERP、财务、采购各管各的，同一个人、同一种药在不同系统里编码对不上，改一处要人工对好几套系统，很累。

MDM 干的事就三件：

- 定一套集团统一的编码和字段标准
- 走 OA 审批，该谁批谁批，别绕过
- 下游系统来这儿拉数据，别各自维护字典

---

## 整体怎么搭的

```
上游（数据从哪儿来）
  OA 审批 ──→ 组织人事（机构/科室/员工）
  供应链   ──→ 药品/耗材/物资
  设备系统 ──→ 设备
  MDM 直录 ──→ 医嘱/规范/诊疗项目
  历史迁移 ──→ 批量导入

         │ A 通道：HTTP 回调（主，秒级）
         │ B 通道：定时轮询（兜底，5 分钟）
         ↓
    主数据平台
      入库校验 → 写库 → 记变更历史
         │
         ↓
下游（数据到哪儿去）
  直接用：HIS/EMR 缓存主数据编码
  对照用：老系统维护 本地编码 ↔ MDM编码 映射表
```

有个硬规矩：**MDM 不是第一个录入的地方**（OA、供应链那边先录），但**数据进了 MDM 之后，下游只能认 MDM**，不能绕过它直接去上游捞。

## 数据怎么进来：A + B 双通道

### A 通道：HTTP 回调（主通道）

OA 或供应链审批过了，上游主动调 MDM 接口写数据：

```
审批通过
  → POST /api/v1/sync/inbound
  → 幂等校验（requestId 防重）
  → 版本号比对（低版本不让盖高版本）
  → 审批单比对（没审批的拦下来告警）
  → 格式/必填/关联校验
  → 过了就写库 + 记变更历史
```

上游调失败了，重试 3 次（5s / 30s / 120s），还不行就丢待同步队列，等 B 通道补。

### B 通道：定时轮询（兜底）

每 5 分钟扫一遍上游增量视图，把 A 通道漏掉的补上：

```sql
SELECT * FROM v_mdm_sync WHERE updated_at > last_sync_time
```

正常情况 B 通道跑完发现没差异，就不写。A 挂了或漏了，B 来擦屁股。

### 冲突怎么处理

1. 比 `version`，低的不能盖高的
2. 停用/注销优先级最高，不会被普通更新意外改回去（比如药已经停了，别又被启用）
3. 版本冲突进异常表，人工看一眼

### 入库校验（不管从哪条通道来，都得过）

```
幂等 + 版本
  → 有没有对应审批单
  → 编码唯一、必填、格式、值域
  → 关联数据在不在（科室挂的机构得存在、医嘱关联的药得存在）
  → 全过才入库，不过进异常表 + 告警
```

## 下游两种用法

| | 直接用 | 对照用 |
|--|--------|--------|
| 适合谁 | 新建 HIS、集团统一部署的系统 | 老分院 HIS、第三方 ERP/财务 |
| 咋干 | 定时拉 MDM 数据到本地缓存，业务单据直接用 MDM 编码 | 本地编码留着，维护一张映射表翻译 |
| 长期 | 推荐最终形态 | 过渡方案，系统升级时再迁 |

对照表主要字段：`mdm_code`、`local_code`、`local_system`、`data_domain`、生效/失效日期。

## 管哪些数据

### 域之间的关系（简图）

```
机构 ─1:N─→ 科室 ─1:N─→ 员工（多点执业走任职记录表）
  │
  ├─ 设备、物资（挂机构）
  ├─ 诊疗项目、规范（N:N 适用机构/科室）
  └─ 医嘱 ─N:N─→ 药品/耗材，N:N 适用科室
```

### 组织人事

**机构**：编码 `ORG` + 6 位流水，树形，状态启用/停用/注销，带 `version`。

**科室**：`DPT` + 8 位，挂机构，也支持树形。

**员工**：`EMP` + 8 位，有主岗机构/科室。多点执业单独一张任职记录表（主岗/兼职/多点执业/进修，带生效失效日期）。

### 供应链

**药品** — 整包 + 拆零两条记录：

- 整包 `DRG00000001`，拆零 `DRG00000001-S`
- 为啥拆两条：HIS 开方、药房发药、收费计价、医保对码，整包拆零是不同 SKU
- 勾选「启用拆零」自动生成子记录；改整包时子记录跟着同步；关拆零就把子记录停用
- 拆零价没填的话，按 `整包价 ÷ 包装数量` 算（4 位小数）

**耗材**：`MAT` + 8 位，高值/低值/试剂，有注册证号和医保编码。

**物资**：`SPL` + 8 位，办公/后勤/维修配件，挂机构。

### 临床

**诊疗项目** `TRT`：检查/检验/治疗/手术等，N:N 适用机构科室，有收费类别和医保编码。

**医嘱** `ODR`：关联药品/耗材/诊疗项目，N:N 适用科室。

**规范** `STD`：临床路径/诊疗指南/操作规程，状态草稿/生效/废止。

### 设备

`DEV` + 8 位，挂机构科室，有资产编号、注册证（医疗设备）、维保到期日。状态：在用/停用/报废/维修中。

### 变更历史（所有域共用一张表）

每次改都记：`data_domain`、`data_code`、改了哪个字段、旧值新值、`version`、审批单号、操作人、来源（OA回调/轮询补偿/人工等）。

### 编码规则

编码不带业务含义，全局唯一，发了就不改、停用也不回收：

| 域 | 格式 | 例子 |
|----|------|------|
| 机构 | ORG + 6位 | ORG000001 |
| 科室 | DPT + 8位 | DPT00000001 |
| 员工 | EMP + 8位 | EMP00000001 |
| 药品整包 | DRG + 8位 | DRG00000001 |
| 药品拆零 | 整包码 + "-S" | DRG00000001-S |
| 耗材 | MAT + 8位 | MAT00000001 |
| 诊疗项目 | TRT + 8位 | TRT00000001 |
| 设备 | DEV + 8位 | DEV00000001 |
| 医嘱 | ODR + 8位 | ODR00000001 |
| 物资 | SPL + 8位 | SPL00000001 |
| 规范 | STD + 6位 | STD000001 |

国药准字、医保编码、ICD 这些是国家标准字段，存在属性里，不当主键用。

## OA 审批

### 通用流程

```
申请人填表 → 部门负责人 → 信息科/主数据组审核 → 分管领导（关键变更）或自动过（小改）
  → 过了：推上游系统 → 回调写 MDM
  → 没过：退回改
```

### 各类申请走哪儿

| 类型 | 审批链 | 先落哪个系统 | 最终进 MDM |
|------|--------|-------------|-----------|
| 机构 | 行政 → 分管副总 → CEO | OA 组织人事 | ✓ |
| 科室 | 院长 → 集团医务部 | OA 组织人事 | ✓ |
| 员工 | 科室主任 → 人力 | OA 组织人事 | ✓ |
| 药品 | 药剂科 → 药事委员会 | 供应链 | ✓ |
| 耗材 | 科室主任 → 采购 → 分管领导 | 供应链 | ✓ |
| 诊疗项目 | 科室主任 → 医务 → 物价 | MDM 直录 | ✓ |
| 设备 | 科室主任 → 设备科 → 分管领导 | 设备系统 | ✓ |
| 医嘱 | 科室主任 → 医务 → 信息科 | MDM 直录 | ✓ |
| 物资 | 部门负责人 → 后勤 | 供应链 | ✓ |
| 规范 | 科室主任 → 医务 → 质控 | MDM 直录 | ✓ |

### 变更类型

| 类型 | 审批力度 |
|------|---------|
| 新增 | 完整流程 |
| 关键字段改 | 完整流程 |
| 非关键字段改 | 部门负责人过就行 |
| 停用 | 完整流程 |
| 注销/合并 | 高级审批 |
| 批量（比如年度药目录调整） | 专项批量审批 |

关键字段举例：机构名/类型/上级、员工证件和主岗、药品通用名/规格/国药准字、设备所属机构等。联系方式、别名、单价这类算非关键。

批量变更：上传 Excel → 系统自动校验 → 生成批量审批单 → 过了批量写入。

## 跟下游系统怎么对接

```
MDM（MySQL + REST API）
  ↕ HTTPS
HIS / EMR / ERP / 财务 / 采购 ...
```

| 方式 | 啥时候用 |
|------|---------|
| HTTP 回调 | 上游审批过了写进来 |
| 增量拉取 | 下游每 15 分钟拉变更 |
| 全量快照 | 规范、医嘱这类低频的，每天凌晨拉一次 |
| 单条查询 | 实时查某条 |

### 集成优先级

| 系统 | 要什么数据 | 模式 | 优先级 |
|------|-----------|------|--------|
| 集团 HIS | 机构/科室/员工/药/耗材/诊疗/医嘱 | 直接用 | P0 |
| 新建分院 HIS | 同上 | 直接用 | P0 |
| 存量分院 HIS | 同上 | 对照 | P0 |
| EMR | 员工/医嘱/规范/诊疗 | 直接用 | P0 |
| ERP | 机构/科室/员工/药/耗材/物资 | 对照 | P1 |
| 财务 | 机构/科室/员工/诊疗/设备 | 对照 | P1 |
| 采购 | 药/耗材/物资 | 对照 | P1 |
| 设备管理 | 机构/科室/设备 | 直接用 | P1 |
| 后勤 | 机构/物资 | 对照 | P2 |

MDM 挂了下游别慌，本地缓存还能撑 24 小时只读。

日常盯这几个数：

| 指标 | 目标 | 频率 |
|------|------|------|
| 完整率（必填非空） | ≥ 99.5% | 每天 |
| 一致率（跨系统比对） | ≥ 99% | 每周 |
| 对照覆盖率 | ≥ 95% | 每周 |
| 同步及时率（审批后 15 分钟内到位） | ≥ 99% | 每天 |

## 技术栈（简版）

模块化单体，一个 JAR 多实例，Nginx 负载均衡。详细版本号和 DDL 看 [tech-doc.md](./tech-doc.md)。

| 层 | 技术 |
|----|------|
| 前端 | Vue 3 + Element Plus + Vite |
| 后端 | JDK 21 + Spring Boot 3.4 |
| 库 | MySQL 9.3（单机，日备 + 小时 binlog） |
| 缓存 | Redis 7（幂等键、热点缓存） |
| 监控 | Prometheus + Grafana |
| 文档 | Knife4j |

### 模块划分

```
mdm-platform/
├── mdm-common        公共工具、异常、配置
├── mdm-auth          登录、角色、API Key
├── mdm-organization  机构/科室/员工
├── mdm-supply        药品/耗材/物资
├── mdm-clinical      诊疗项目/医嘱/规范
├── mdm-facility      设备
├── mdm-sync          入站校验、出站分发、对照映射
├── mdm-quality       质量检查、报表
├── mdm-audit         变更历史、操作日志
└── mdm-app           启动入口
```

模块间用 Spring Event 通信，事务本地 `@Transactional` 搞定。

### 备份

- MySQL：日全量 + 小时 binlog，保留 30 天
- 变更历史表：不删，永久留
- 应用 2 实例，Actuator 健康检查，Nginx 摘故障节点
- Redis 挂了自动降级直查库

## 以后加新数据域咋扩

现在 10 个域，以后可能加床位、收费项目、医保目录、病区、手术目录等。

靠 `mdm_domain_registry` 注册表驱动，同步/校验/质量模块按 `domain_code` 路由，不用改框架：

```
注册表里配：domain_code、表名、编码前缀、要不要审批、要不要质量检查
  → 建表（遵循统一字段规范：code、name、org_code、status、version、审计字段、deleted）
  → 实现 DomainSyncHandler
  → 前端加页面和菜单
```

新表必备字段：`id`、`{domain}_code`（唯一）、`{domain}_name`、`org_code`、`status`、`version`、创建/更新人时间、`deleted`。

### 数据量预估（30 家机构）

主数据总共不到 10 万条，MySQL 单机够用。变更历史增长快，一年 10 万+，以后可以先拆历史库，极端情况再按域分库——模块本来就按域隔离，改数据源配置就行，业务代码不用动。

## 分期计划

**一期（1-3 月）**：框架 + 机构科室上线，对接 OA 和集团 HIS，历史数据并行清洗。

**二期（4-6 月）**：员工 + 药品/耗材/物资，对接供应链/ERP/采购，质量监控上线。

**三期（7-9 月）**：诊疗/医嘱/规范/设备，对接 EMR 和设备系统，1-2 家分院试点后推广，批量建对照表。

**四期（10-12 月）**：全集团接入、驾驶舱、运维体系、应急演练。

| 节点 | 时间 | 验收 |
|------|------|------|
| 机构科室可用 | M3 | HIS 能同步机构科室 |
| 员工+供应链 | M6 | 药/耗材/员工统一管理 |
| 临床+设备 | M9 | EMR 对接，分院试点完 |
| 全面上线 | M12 | 下游全接入，质量指标达标 |

## 挂了怎么办

| 情况 | 咋办 |
|------|------|
| MDM 服务挂了 | 下游用本地缓存撑（24h），恢复后拉全量快照补齐 |
| 上游回调失败 | B 通道 5 分钟内补 |
| 数据库挂了 | 友好报错，下游走缓存，从备份恢复后验数据 |
| 校验大面积拦截 | 正常的继续入库，异常的进异常表人工处理 |

上线按机构灰度，先 1-2 家分院跑 1-2 周。支持按变更历史回滚指定域的数据。应用蓝绿/滚动发布。

应急分级：P0 马上降级修，P1 四小时内，P2 二十四小时内，修完复盘。

## 主要风险

| 风险 | 应对 |
|------|------|
| 历史数据烂，迁移费劲 | 先摸底清洗，分批迁，不挡功能上线 |
| 下游改造慢 | 提供对照模式 + 标准 API，按优先级分批接 |
| 上游绕过审批改数据 | B 通道检出 + 入库比对审批单 |
| OA 流程推不动 | 非关键字段简化审批，循序渐进 |
| 标准难统一 | 参照国标，专家评审，编码不带业务含义 |
| MDM 自身故障 | 下游缓存降级、双实例、定期演练 |

## 做完有啥用

- 主数据准确率从不到 80% 拉到 99%+，脏数据入库前就拦
- 跨系统对账工作量能少一大截
- 变更全留痕，审计查得着
- 新系统对接成本低，调 API 就行
- 没审批的变更、乱改状态能被自动发现
